## 1. TL;DR

The Jun-9 drop is **not an evolution of the current landing page** — it is a wholesale pivot from a *one-page marketing site* ("Package AI. Ship Everywhere.") to a *community-hub dashboard product* ("stevedores-org · community hub") with a sidebar shell, six top-level views, charts, an interactive terminal, and a container-list simulator.

- **Brand surface area:** narrative shifts from *"AI agent packaging platform"* → *"open-source container community"* (Docker-adjacent). The word "AI" does not appear in the new design's copy.
- **UX surface area:** grows from ~1 hero page → 6 navigable views (dashboard / containers / terminal / projects / contributors / activity).
- **Engineering surface area:** grows from React + Tailwind (zero runtime deps beyond React 19) → React + Tailwind v4 + **40+ shadcn/ui primitives** + recharts + lucide-react + motion + react-hook-form + sonner + vaul + cmdk + others.
- **Data:** every view is hard-coded mock data. No API contracts exist in the design. Wiring to a real backend is **out of scope** for the design and must be specced separately.

**Recommendation:** Treat this as a v2 site rebuild. Do **not** merge as a drop-in replacement for `src/App.tsx`. Stand up the design behind a route or sub-domain first, validate the positioning shift with stakeholders, then decide migration path.

### 4.3 Dependency footprint (net adds if we merge as-is)

The current `package.json` declares: `react`, `react-dom`. The design adds **~50 runtime dependencies**, including the entire Radix UI set, recharts, motion, sonner, cmdk, vaul, react-hook-form, embla-carousel, react-day-picker, date-fns, lucide-react, tailwind-merge, class-variance-authority, tw-animate-css, canvas-confetti, input-otp, react-dnd, react-popper, react-router, react-slick, react-responsive-masonry, react-resizable-panels, next-themes, plus MUI `@mui/material` + `@mui/icons-material` (likely vestigial — not actually imported in the source files inspected).

Risks introduced:
- **Bundle size** jumps from ~150 KB to multi-MB range without code-splitting strategy.
- **React 18 vs 19**: design declares `react: 18.3.1` as a peer dep; repo is on `react: ^19.2.0`. Likely compatible (Radix and shadcn primitives support React 19) but must be verified.
- **MUI in the manifest, shadcn in the code:** dead-weight unless cleaned up.

## 5. Impact assessment

### 5.1 Stakeholder / strategic
- **High-impact:** the design **drops "AI"** from the visible positioning. The repo's mission (per `README.md` and `AGENTS.md`) is centered on AI agent packaging for Lornu AI. Before adoption, get explicit confirmation that the public face of stevedores.org is intentionally pivoting away from AI framing toward a generic "container community" story. Possible reasons:
  1. **Intentional split** — stevedores.org is being repositioned as the OSS community front, while the AI-packaging product lives under a different brand (lornu.ai / dockworker.ai / etc.).
  2. **Stale design brief** — designer was given a "Docker community hub" framing without the current AI positioning context. In which case the design needs a copy pass.
- The six referenced repos (`stevedore`, `harbor-cli`, `dock-compose`, `sandbox-runner`, `cargo-runner`, `portainer-lite`) **do not appear to exist** under `stevedores-org` today. Either (a) they need to be created/forked, (b) the design's data is placeholder we'll swap, or (c) the design predicts a roadmap that hasn't been decided.

### 5.2 Engineering effort estimate (rough — sized for a 1-FE-engineer-week unit)

| Phase | Work | Estimate |
|---|---|---|
| **Phase 0** — clarify positioning, get sign-off on pivot vs. coexistence | non-engineering | blocks all later work |
| **Phase 1** — port design into repo behind `/preview` route, no backend; React 19 + Tailwind v4 reconciliation; dead-MUI removal; replace `<title>` and remove `noindex` | 1 FE-week |
| **Phase 2** — replace static landing at `/` with new shell *or* publish at a sub-route; resolve `src/App.tsx` vs `frontend/server.ts` duality | 0.5 FE-week |
| **Phase 3** — wire dashboard / projects / contributors / activity to **real GitHub data** (cached via an edge worker or Bun-Hono server) | 2 FE-weeks |
| **Phase 4** — terminal & container simulator: **decide scope** — keep as static demos? front a real `stevedore` daemon? expose a sandboxed shell? Each option has different security/infra cost | 1–6 weeks depending on scope |
| **Phase 5** — channels: pick chat backend, embed, or drop section | 0.5–2 weeks |
| **Phase 6** — accessibility / responsive pass (the design appears desktop-only; small font sizes will likely fail WCAG AA) | 0.5 FE-week |

**Order-of-magnitude:** **~5 FE-weeks for a credible v1**; **10–14 weeks if terminal/sandbox is in scope and chat is real.**

### 5.3 Risk register

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | Brand pivot ships before stakeholders agree | High | Park new design behind `/v2` until positioning is approved |
| R2 | Bundle/load regression on `/` | Med | Code-split per view; lazy-load recharts; tree-shake icons |
| R3 | Terminal sandbox interpreted as a real shell — security expectation mismatch | High (if shipped as-is, users will type real credentials into a mock) | Add clear "demo / read-only" banner; consider removing for v1 |
| R4 | Repo names referenced in design don't exist under `stevedores-org` | Med | Either reserve those names on GitHub or swap to real repo list before publishing |
| R5 | Accessibility — 0.55rem font is ~9px | Med-High | Bump density tokens; verify contrast on `#5a7a5a` vs `#060d06` (likely fails) |
| R6 | `MARKER-MAKE-KIT-INVOKED` and "Design Review" `<title>` + `noindex` leak into prod | Low | Pre-merge checklist |
| R7 | Two competing site sources (`src/App.tsx` vs `frontend/server.ts`) — adding a third is worse | Med | Reconcile first |

### 5.4 What we can adopt cheaply (low-risk wins)
- The **dark-green palette** as an additional brand option (token-only change), even if we keep the marketing hero.
- The **monospace typography** for code-adjacent surfaces.
- The **shadcn primitive set** as a vendored UI library — useful regardless of which design we ship.
- The **stat-card row** + **commit chart** pair: a "live OSS-org telemetry" widget on the existing landing page, no rebuild required.

## 6. Release / rollback strategy

To make this design **frozen, named, and re-pullable** after future iterations, the original zip is published as a GitHub Release on this repo:

- **Tag scheme:** `design-vYYYY.MM.DD` (e.g. `design-v2026.06.09`) — date-based and sortable; designs are not semver-ish, so the date keys the iteration. Add `-N` suffix only if multiple drops land on the same day (e.g. `design-v2026.06.09-2`).
- **Tagged commit:** the commit that adds *this* gap-map under `docs/design/`. The repo tree at that commit is the rollback target if a design experiment goes off the rails.
- **Release asset:** the original zip — uploaded verbatim so the design can be re-extracted bit-for-bit, regardless of any later edits made when porting into code. The release is **prerelease + non-latest** so it never overrides "Latest release" surfaces for code.
- **Rollback drill:** to re-pull a prior design — `gh release download design-v2026.06.09 -R stevedores-org/stevedores.org -p '*.zip'` → unzip → diff against current. To revert a port that has gone wrong — `git revert` the port commits; the tag itself is immutable and serves as the always-correct reference.

This is the **first** release of any kind on this repo (no prior tags), so this drop establishes the convention. The same pattern should be followed for the next drop.
